## Why

When a new version of nyt-scorebot is deployed to the Raspberry Pi, there is no signal in Discord that anything changed. Players have no visibility into new features, fixes, or behaviour changes, and the team has no incentive to write meaningful commit messages. Automating release-note generation and posting closes that loop without adding manual overhead.

## What Changes

- Adopt **conventional commit** prefixes (`feat:`, `fix:`, `refactor:`, `chore:`, `ci:`, `test:`, `docs:`) and document them in `.github/copilot-instructions.md` so Copilot follows the convention when it drafts commits.
- Add an AI-driven release-notes generation step to the pipeline that calls **GitHub Models** (using the built-in `GITHUB_TOKEN` and `models: read` permission) to summarise commits since the last tag into concise, user-facing notes. Housekeeping commits (`chore`, `ci`, `test`) are filtered out by the prompt.
- **Reorder the pipeline** so `release` runs **before** `deploy`, ensuring the GitHub Release exists (with AI-generated notes as its body) when the bot restarts on the Pi. This is a change from the current order (`deploy → release`).
- Add a new `announce` job after `deploy` that POSTs a formatted message to a dedicated Discord release-notes channel via a **Discord webhook URL** stored as a GitHub secret. The message includes the version, a concise bullet list, and a link back to the full GitHub Release.
- Add a new repository secret: `DISCORD_RELEASE_WEBHOOK_URL`. No bot code changes; no exposure of `DISCORD_TOKEN` to the announce job.
- Update the existing `release-creation` and `pipeline-orchestration` specs to reflect that the release workflow is part of the pipeline (drift already exists in the implementation) and that release notes are AI-generated rather than auto-generated by GitHub.

## Capabilities

### New Capabilities
- `discord-release-announcement`: Posts a formatted release-notes message to a dedicated Discord channel via webhook after a successful deployment, linking to the full GitHub Release.
- `ai-release-notes`: Generates release-notes content from commit history using GitHub Models, filtering housekeeping commits and producing a concise bullet list suitable for a Discord audience.
- `commit-message-conventions`: Project-wide convention requiring conventional-commit prefixes, documented in `.github/copilot-instructions.md` so AI assistants follow it.

### Modified Capabilities
- `release-creation`: The release workflow is no longer independent of the pipeline and no longer relies on GitHub's `generate_release_notes`. It is now invoked by the pipeline before deploy and uses AI-generated notes as the release body.
- `pipeline-orchestration`: Pipeline order changes from `build → test → deploy` to `build → test → release → deploy → announce`. Release becomes a required pipeline stage rather than a manual side workflow.

## Impact

- **Workflows changed**: `.github/workflows/pipeline.yml` (reorder + new `announce` job), `.github/workflows/release.yml` (AI notes generation, called from pipeline), new `.github/workflows/announce.yml` (Discord webhook post).
- **New secret required**: `DISCORD_RELEASE_WEBHOOK_URL` (created via Discord Server Settings → Integrations → Webhooks for the dedicated release-notes channel).
- **New permission required**: `models: read` on the release job in the workflow.
- **Documentation**: `.github/copilot-instructions.md` updated with conventional commit guidance.
- **No application code changes**, no new runtime dependencies, no schema changes, no impact on the Pi.
- **External dependencies**: GitHub Models inference endpoint (`models.github.ai`) and Discord webhook endpoint must be reachable from the GitHub-hosted runner (they are, by default).
